# 音色链路

## 1. 目标

通过长期负责音色链路（训练 + 接入 + 稳定性）。


## 2. 工作边界

可改文件：
- `tts/base_tts.py`
- `tts/edge.py`
- `tts/sovits.py`
- `tts/qwentts.py`
- `tts/indextts2.py`

不直接改文件：
- `avatars/*`
- `streamout/*`
- `server/*`
- `app.py`
- `config.py`
- `registry.py`

如确需改公共文件，先提交变更说明，经确认后再改。


## 3. 长期任务清单

## A. 音色训练线

### A1. 数据建设
要做：
1. 持续扩充高质量人声数据（干净、单人、少混响）。
2. 维护版本化数据清单（时长、句子数、采样率、文本覆盖）。
3. 建立“坏样本剔除规则”（爆音、噪声、重叠说话、错标文本）。

预期效果：
- 训练数据质量可控，模型质量随数据迭代稳定提升。

### A2. 训练迭代
要做：
1. 按固定模板记录训练参数（学习率、batch、epoch、数据版本）。
2. 每轮训练产出可追踪模型版本（模型名、时间、配置摘要）。
3. 形成“继续训练 / 停止训练 / 回滚旧版”的结论。

预期效果：
- 每一版音色模型都可复现、可比较、可回滚。

### A3. 主观 + 客观评估
要做：
1. 维护固定测试文本集（短句、长句、数字、英文混读、情绪句）。
2. 每版输出试听样例并打分（清晰度、自然度、稳定性）。
3. 记录明显问题（吞字、抖动、拖尾、噪声、语气异常）。

预期效果：
- 模型优劣有证据，不靠主观印象拍脑袋。


## B. 工程接入线

### B1. 事件一致性
要做：
1. 所有 TTS 插件统一 `start/end` 行为。
2. 确保中断后不继续输出旧句音频。
3. 保证状态事件与文本对应关系不乱序。

预期效果：
- 上层业务能稳定感知每句播报生命周期。

### B2. 流式边界稳定性
要做：
1. 统一处理 chunk 尾包和残留样本。
2. 避免末句丢失、重复、卡顿。
3. 长文本连续播报时保持节拍稳定。

预期效果：
- 实时播报平滑，长句不崩，不乱节奏。

### B3. 可观测性
要做：
1. 在 `base_tts.py` 统一日志字段（插件、文本、状态、错误）。
2. 各插件按同一格式输出。
3. 出错时日志能定位到具体阶段（请求、解码、重采样、下发）。

预期效果：
- 排障时间明显下降，问题可快速归因。


## C. 文档与交付线

### C1. 自测文档
要做：
1. 维护可执行的自测步骤。
2. 覆盖短句、长句、中断、插件切换。
3. 每次行为变化时同步更新文档。

预期效果：
- 其他同事可独立复测，不依赖口头交接。

### C2. 变更说明
要做：
1. 每次提交说明“改了什么、为什么改、影响范围”。
2. 若涉及行为变化，说明兼容策略。
3. 给出回滚办法。

预期效果：
- 对接成本低，复盘成本低。


## 4. 关键验收门槛

1. `python app.py --model wav2lip --transport webrtc` 可启动。
2. 连续 20 句播报，每句都有 `start/end`。
3. 调用 `/interrupt_talk` 后无旧句残留。
4. `edgetts` 与 `gpt-sovits` 可切换且不崩。
5. 日志可追踪每句从输入到结束的状态流转。
6. **代码提交一定要标注详细的中文注释，方便维护。**
7. **需要适配多语言直播平台，外语目前只适配英语平台。**


## 5. 长期交付物（持续维护）

1. 模型版本清单（含训练参数摘要）。
2. 训练数据版本清单。
3. 固定测试集试听结果记录。
4. 自测步骤文档（持续更新）。
5. 问题复盘清单（问题、原因、修复、预防）。
